- 这台机器上：
  - 普通网页访问 GitHub 正常
  - GitHub REST API 正常
  - 但 `git remote-https` 的 smart-HTTP 传输会在认证后被重置
- 所以问题不在仓库权限，而在 Git 这条传输链路本身

### 我们做过的验证

- 检查了 `github.com:443`，网络端口是通的
- 检查了 Git 配置、凭据和代理线索
- 打开了 Git 的详细网络日志
- 确认了：
  - `origin` 配置正确
  - GitHub 凭据可用
  - REST API 可以正常访问仓库
  - 失败点主要发生在 Git 的 smart-HTTP 请求阶段

### 最终处理方式

- 本地先正常提交了今天的笔记
- 当标准 `git push` 失败后，改用 GitHub API 把文件内容同步到了远程仓库
- 为了减少后续混乱，把本地工作分支统一整理为 `main`
- 同时保留一个本地历史备份分支：`codex/master-pre-main-sync`

### 当前结论

- 远程仓库已经有今天的笔记文件

- 但由于远程首次写入是通过 API 完成的，本地和远程 Git 历史目前不是完全同构
- 这不影响我们继续学习和开发，但要记住这是一笔“已知技术债”

### 为什么要把这件事记下来

- 以后重新开线程时，不会误以为“push 失败是权限问题”

## Session 009 - Git 调试与 SSH over 443 固化

日期：2026-03-12

### 我们这一步在做什么

把今天反复出现的 GitHub 推送不稳定问题彻底查清楚，并把更稳的连接方案落到本机：

- 检查 Git 代理、环境变量代理、WinHTTP 代理
- 检查 DNS 和 `github.com:443` 连通性
- 分别验证普通 HTTPS、`git ls-remote`、`git push`
- 最后切换到 `SSH over 443`

### 这一步学到的东西

- 问题并不像“代理配错”那么简单
- 普通 HTTPS 访问 GitHub 和 `git ls-remote` 可以正常，不代表 `git push` 一定稳定
- 本机更像是对 GitHub HTTPS push 这条链路存在偶发连接重置
- 对这台机器来说，`SSH over 443` 比 HTTPS push 更适合作为默认方案

### 我们做过的排查

- 确认 Git 没有额外的 `http.proxy` / `https.proxy`
- 确认环境变量里没有代理残留
- 确认 WinHTTP 是 direct access
- 确认 `github.com:443`、`github.com:22`、`ssh.github.com:443` 都可以探测
- 抓了 `git push` 的详细日志，确认认证和收发包流程本身能走通，但 HTTPS push 仍有偶发 reset

### 最终处理方式

- 在 `C:\\Users\\Administrator\\.ssh` 下生成专用 GitHub SSH key
- 配置 `C:\\Users\\Administrator\\.ssh\\config`
- 让 `github.com` 通过 `ssh.github.com:443` 建立 SSH 连接
- 把仓库 `origin` 从 HTTPS 改成 `git@github.com:Tiyou-zm/myagent_elf.git`
- 验证 `ssh -T git@github.com` 成功
- 验证 `git push origin main` 成功

### 当前结论

- 这台机器后续应优先使用 SSH 推送 GitHub
- 如果再次遇到推送问题，优先检查 SSH 连通性，而不是先怀疑业务代码
- 今天这次排障结果已经被固化到项目文档，后续不需要从头再查一遍

## Session 010 - 索引主流程与切块机制学习

日期：2026-03-13

### 我们这一步在做什么

今天没有继续加新功能，而是沿着现有代码把“索引到底怎么跑起来”这条链路拆开学清楚：

- 先看 `build_index()` 是怎么调度整条索引流程的
- 再看文件没变为什么可以跳过
- 然后看文件一旦变化，为什么是整文件重切重写
- 最后把 `build_chunks()` 的切块规则讲明白

### 这一步学到的东西

- `build_index()` 的主骨架是：
  - 遍历候选文件
  - 读取文件状态
  - 查旧记录
  - 没变就跳过
  - 变了就读正文、切块、入库
- 当前增量索引判断依赖 `size_bytes + mtime_ns`
- 这里不直接用 `content_hash` 做第一层判断，是因为算 hash 之前就得先把文件读出来，成本更高
- 当前策略不是块级增量更新，而是文件级判断后整文件替换文本块

### 关于切块的理解

- 切块不是按行号切，也不是按句子切
- 主规则是按字符数控制块大小
- 但实现时是一行一行往当前块里加，避免把单行内容硬切断
- 所以块大小是近似控制，可以略微超过阈值
- 行号的作用不是决定怎么切，而是给切好的块做定位标签

### 关于重叠内容的理解

- 一块切完后，会保留块尾的一小段内容给下一块继续用
- 当前重叠大小按字符数配置，而实际保留时按行回收
- 这样做是为了让前后块之间的上下文更连贯，避免命中结果断得太厉害
- 也正因为下一块不是从空白开始，所以要重新计算：
  - 当前块字符数
  - 当前块起始行号

### 这一步形成的关键结论

- 当前索引策略可以概括成一句话：
  - 文件没变就跳过，文件变了就整篇重切重写
- 当前切块策略可以概括成一句话：
  - 按字符数控大小，按行保完整，按重叠保上下文

### 下一步

- 后续可以继续往搜索阶段推进
- 重点看切好的块在 FTS5 搜索里是怎么被命中、打分和返回片段的

- 真正的内容搜索发生在 `storage.py` 的 SQL 里
- 内容搜索不能从 `files` 表开始，因为 `files` 表不保存正文全文索引
- 内容搜索真正搜的是“命中的文本块”，再通过文本块回查所属文件

### 关于结果字段的理解

- `snippet(...)` 的作用是从命中的文本块里截一段更适合展示的片段
- `bm25(...)` 的作用是给每条结果计算相关度分数，用来排序
- `snippet` 偏向展示给人看
- `chunk_index`、`start_line`、`end_line` 偏向给系统和定位使用

### 这一步形成的关键结论

- 当前内容搜索可以概括成一句话：
  - 先找命中的块，再回头找到这个块属于哪个文件
- 当前结果返回可以概括成一句话：
  - 既要让人看懂命中了什么，也要让系统知道命中在哪里

### 下一步

- 继续往搜索阶段推进
- 可以接着看 API 层是怎么把这些搜索结果返回给前端的
