## Session 001 - 项目立项

日期：2026-03-12

### 我们这一步在做什么

我们没有急着做一个“很强的 Agent”，而是先把目标收敛到一个最小可用版本：

- 做一个带桌面宠物外壳的个人助理
- `v1` 先专注于本地文件检索

### 这一步学到的东西

- `Agent` 不等于“一个会聊天的模型”
- 一个可用的 Agent 至少要有：
  - 明确目标
  - 可调用的工具
  - 可追溯的数据来源
  - 清楚的边界
- 桌面宠物只是表现层，不是核心能力本身
- 真正有价值的能力是：理解意图、找到信息、把结果可靠地交给用户

### 我们做出的关键决定

- `v1` 先不碰密码保存
- `v1` 不扫描整个电脑
- `v1` 只处理用户指定目录
- 技术路线先定为：
  - Electron 做桌面端
  - Python 做 Agent 服务
  - SQLite 做本地存储和检索基础

### 为什么这些决定重要

- 先收窄范围，项目更容易真正做完
- 先解决“找得到文件”这个刚需，比先做花哨交互更有价值
- 安全边界一开始就立住，后面不容易走偏

- 已初始化 Git 仓库

## Session 003 - Git 推送排障与历史整理

日期：2026-03-12

### 我们这一步在做什么

把今天的学习记录真正同步到 GitHub，同时把本地仓库的分支状态整理成后面可继续开发的样子。

### 这一步学到的东西

- “能和大模型对话”不等于“本机 Git 到 GitHub 的传输一定正常”
- 这台机器上：
  - 普通网页访问 GitHub 正常
  - GitHub REST API 正常
  - 但 `git remote-https` 的 smart-HTTP 传输会在认证后被重置
- 所以问题不在仓库权限，而在 Git 这条传输链路本身

### 我们做过的验证

- 检查了 `github.com:443`，网络端口是通的
- 检查了 Git 配置、凭据和代理线索
- 打开了 Git 的详细网络日志
- 确认了：
  - `origin` 配置正确
  - GitHub 凭据可用
  - REST API 可以正常访问仓库
  - 失败点主要发生在 Git 的 smart-HTTP 请求阶段

### 最终处理方式

- 本地先正常提交了今天的笔记
- 当标准 `git push` 失败后，改用 GitHub API 把文件内容同步到了远程仓库
- 为了减少后续混乱，把本地工作分支统一整理为 `main`
- 同时保留一个本地历史备份分支：`codex/master-pre-main-sync`

### 当前结论

- 远程仓库已经有今天的笔记文件
- 本地仓库也保留了完整内容
- 但由于远程首次写入是通过 API 完成的，本地和远程 Git 历史目前不是完全同构
- 这不影响我们继续学习和开发，但要记住这是一笔“已知技术债”

### 为什么要把这件事记下来

- 以后重新开线程时，不会误以为“push 失败是权限问题”
- 我们能快速回忆起这台机器的 Git 特性和已经验证过的路径
- 后续如果网络环境改善，可以再把 Git 历史进一步收敛
