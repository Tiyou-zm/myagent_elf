## Session 001 - 项目立项

日期：2026-03-12

### 我们这一步在做什么

我们没有急着做一个“很强的 Agent”，而是先把目标收敛到一个最小可用版本：

- 做一个带桌面宠物外壳的个人助理
- `v1` 先专注于本地文件检索

### 这一步学到的东西

- `Agent` 不等于“一个会聊天的模型”
- 一个可用的 Agent 至少要有：
  - 明确目标
  - 可调用的工具
  - 可追溯的数据来源
  - 清楚的边界
- 桌面宠物只是表现层，不是核心能力本身
- 真正有价值的能力是：理解意图、找到信息、把结果可靠地交给用户

### 我们做出的关键决定

- `v1` 先不碰密码保存
- `v1` 不扫描整个电脑
- `v1` 只处理用户指定目录
- 技术路线先定为：
  - Electron 做桌面端
  - Python 做 Agent 服务
  - SQLite 做本地存储和检索基础

### 为什么这些决定重要

- 先收窄范围，项目更容易真正做完
- 先解决“找得到文件”这个刚需，比先做花哨交互更有价值
- 安全边界一开始就立住，后面不容易走偏

- 已初始化 Git 仓库
- 已建立共同笔记文件

## Session 003 - Git 推送排障与历史整理

日期：2026-03-12

### 我们这一步在做什么

把今天的学习记录真正同步到 GitHub，同时把本地仓库的分支状态整理成后面可继续开发的样子。

### 这一步学到的东西

- “能和大模型对话”不等于“本机 Git 到 GitHub 的传输一定正常”
- 这台机器上：
  - 普通网页访问 GitHub 正常
  - GitHub REST API 正常
  - 但 `git remote-https` 的 smart-HTTP 传输会在认证后被重置
- 所以问题不在仓库权限，而在 Git 这条传输链路本身

### 我们做过的验证

- 检查了 `github.com:443`，网络端口是通的
- 检查了 Git 配置、凭据和代理线索
- 打开了 Git 的详细网络日志
- 确认了：
  - `origin` 配置正确
  - GitHub 凭据可用
  - REST API 可以正常访问仓库
  - 失败点主要发生在 Git 的 smart-HTTP 请求阶段

### 最终处理方式

- 本地先正常提交了今天的笔记
- 当标准 `git push` 失败后，改用 GitHub API 把文件内容同步到了远程仓库
- 为了减少后续混乱，把本地工作分支统一整理为 `main`
- 同时保留一个本地历史备份分支：`codex/master-pre-main-sync`

### 当前结论

- 远程仓库已经有今天的笔记文件
- 本地仓库也保留了完整内容
- 但由于远程首次写入是通过 API 完成的，本地和远程 Git 历史目前不是完全同构
- 这不影响我们继续学习和开发，但要记住这是一笔“已知技术债”

### 为什么要把这件事记下来

- 以后重新开线程时，不会误以为“push 失败是权限问题”
- 我们能快速回忆起这台机器的 Git 特性和已经验证过的路径
- 后续如果网络环境改善，可以再把 Git 历史进一步收敛

## Session 004 - 网络恢复后完成标准 Git 对齐

日期：2026-03-12

### 我们这一步在做什么

在切换网络环境后，重新验证 GitHub 链路，并把本地与远程仓库重新拉回标准 Git 工作流。

### 这一步学到的东西

- 是否“能上网”并不重要，关键是 `git fetch / git push` 这条链路是否真的恢复
- 一旦 `ls-remote` 和 `fetch` 能稳定成功，就可以开始做真正的历史对齐
- 对齐历史前先看文件内容差异，避免误把“只有换行不同”当成真实内容冲突

### 这一步的验证结果

- `git ls-remote` 成功
- `git fetch origin main` 成功
- 本地和远程文件内容差异确认只有换行层面的区别
- 这说明现在可以安全地把本地工作分支重新对齐到远程 `main`

### 最终处理方式

- 保留已有备份分支：`codex/master-pre-main-sync`
- 继续以 `main` 作为本地工作分支
- 用标准 Git 把本地 `main` 对齐到远程 `origin/main`
- 再用一次标准 `git push` 验证链路恢复

### 当前结论

- 这次 Git 问题已经从“只能 API 兜底”推进到了“标准 Git 工作流恢复”
- 后续正常开发、提交、推送都应优先走标准 Git
- 之前那段 API 兜底历史仍然被记录在日记里，作为一次真实的排障案例保留

### 下一步

开始搭 `Python` 索引服务骨架，并把第一版目录结构和 API 设计也同步记进这份日记。

## Session 005 - Python 索引服务骨架落地

日期：2026-03-12

### 我们这一步在做什么

把仓库从“只有项目笔记”推进到“有第一版可扩展后端骨架”，目标不是一次做全，而是先把最小链路真正立起来：

- 扫描用户指定目录
- 抽取文本内容
- 写入 SQLite FTS5 索引
- 提供最小查询 API

### 这一步学到的东西

- “骨架”不是只画目录树，而是要让关键职责先分层站住
- 对这个项目来说，最先稳定的不是 Agent 编排，而是检索底座
- 如果现在就把扫描、存储、搜索混写在一起，后面加摘要、重排、语义检索会很难拆

### 这一步做出的结构决定

- 包目录定为：`src/index_service`
- 当前按 4 层拆开：
  - `api.py`：HTTP 接口
  - `indexing.py`：扫描、读取、切块、建索引
  - `storage.py`：SQLite/FTS5 持久化
  - `search.py`：查询服务
- 配置集中在 `config.py`
- 先只支持常见文本文件，避免一开始就陷入复杂解析兼容性

### 当前最小接口

- `GET /healthz`
- `POST /api/v1/index`
- `POST /api/v1/search`

### 当前仍然刻意不做的事

- 不做整机扫描
- 不做 Office/PDF 等复杂文档解析
- 不做 embedding 和语义召回
- 不做桌面端联调

### 为什么这样推进

- 这样能最快验证“本地文件检索助理”最核心的价值链路
- SQLite FTS5 足够支撑第一版可演示能力
- 后面无论接 Electron，还是增加摘要/语义检索，都有稳定底座可挂接

### 当前结果

- 已补第一版 Python 服务目录
- 已补依赖清单和本地运行说明
- 已补一个覆盖“建索引 -> 搜索”的最小测试

### 下一步

- 先验证 FastAPI 依赖安装和接口可启动
- 再补增量更新、忽略规则、结果排序
- 然后再考虑 Electron 侧如何调用这些接口

## Session 007 - 补增量索引、文件检索和根目录查询

日期：2026-03-12

### 我们这一步在做什么

在第一版骨架之上，继续把它往“更像能用的 v1”推进，而不是只停在演示级骨架：

- 给索引流程补最小增量能力
- 给服务补文件名检索
- 给服务补已登记根目录查询

### 这一步学到的东西

- “能建索引”不等于“每次都该从头重建索引”
- 对本地文件检索助理来说，找“文件名”本身就是一条独立需求，不只是找内容
- 目录授权边界除了输入时要控制，服务里也应该能回看当前登记状态

### 这一步做出的实现调整

- 在索引时先读取已有文件元数据
- 如果文件大小和修改时间都没变，就先跳过重建文本块
- 新增基于路径模糊匹配的文件检索
- 新增根目录列表查询接口

### 当前接口

- `GET /healthz`
- `GET /api/v1/roots`
- `POST /api/v1/index`
- `POST /api/v1/search`
- `POST /api/v1/search/files`

### 当前结果

- 第二次索引同一批未变化文件时，会走跳过逻辑
- 服务现在既能按内容找，也能按文件名找
- 测试已覆盖：
  - 内容建索引与搜索
  - 增量索引跳过
  - 文件名检索
  - 根目录登记查询

### 下一步
